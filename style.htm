Getting Started with HTML & CSS

1.6: CSS Basics

Learning Goals

  • Explain CSS syntax as well as how to style a basic webpage
  • Compare inline styling and external stylesheets

 Estimated Read Time: 1 Hour.

Introduction

Welcome back! In the previous Exercises, you received an introduction to HTML, and you also began building your own portfolio website, a project you'll continue to develop throughout the rest of this course. In addition, you explored the nitty-gritty aspects of setting up a project in GitHub, which will come in handy for nearly all of your design and development projects in the future.

Now that the basic layout of your webpage is in place, how can we inject some style into it, adding that unique-yet-user-centric flair to your portfolio? The magic wand that defines your website’s layout, color, font size, and typography is called CSS. You might be familiar with this term already, as it’s mentioned often by designers and developers alike. While it’s not a programming language itself, per se, it is an important language you can (and should) use alongside HTML.

In this Exercise, we'll add CSS to your knowledge base, giving your portfolio website a personalized and memorable identity. Let's jump right in!

What is CSS?

CSS, which stands for Cascading Style Sheets, gives you the power to style the look and feel of a webpage. It’s a language that controls the presentation of an HTML file by defining typography, color, layout, and more.

CSS was introduced in its first version (level 1) in 1997 to separate the content and structure (HTML) of a website from the styling (CSS). In the beginning, CSS level 1, also known as CSS1, was restricted to basic styling elements such as color and spacing. As the language continued to evolve, so did its ability to handle more elaborated styling. Today, CSS3 allows you to build everything from layouts and gradients to short animations, and nearly all of its functionality is supported by every modern browser on the market.

TIP!
In case you’re wondering if there’s a CSS4 coming soon, know that there’s no official version in the works at the moment. There is, however, a CSS working group, sometimes referred to as CSS WG, constantly working on new features for CSS. While some of these features will, indeed, be available in a level 4 version, the current term used throughout the industry when discussing state-of-the art CSS is CSS3.

What Does Cascading Style Sheets Mean?

The term stylesheet, here, is mostly self-referential. A stylesheet is essentially a “sheet” containing a great many “styles” used to define the look and feel of a webpage. The special part of the naming comes in the form of the word cascading, which, in this context, describes how each style in the stylesheet is applied to an HTML document.

Every webpage is “styled” by at least one stylesheet—the default stylesheet of the browser. But a webpage can also have more than one stylesheet. No matter how many stylesheets a webpage has, the HTML document will “listen” to every one of them. How, then, does it know which styles have priority? This is where the “cascading” part comes in!

Rules defined further down in a stylesheet will overwrite rules specified on top. This means that each new rule you create should be more specific than the ones preceding it (changing only one or two styling elements, for example, rather than defining the entire style of an element). As the term suggests, CSS rules are statements that tell browsers how to display certain elements on an HTML page, but what does a styling rule look like? Let’s begin by taking a look at CSS syntax.

CSS Syntax

We’ve just learned that we can style our website with CSS rules. Now, we need to figure out what these CSS rules (also called rule-sets) look like and how they work. Let’s say you want to give your portfolio site a light gray background color. You’ll do this by writing your first CSS rule. Open a new file in your text editor and save it as “style.css” in your CSS folder.

First, you need to select the HTML element you want to style. This part of the rule-set is called the selector. As you want to set the background color of the whole site, you select the body element. One selector can have one or multiple stylings called declarations. The stylings are grouped within curly brackets. Each styling consists of a property, which is the part you want to style (background-color), and a value, which defines how you want to style the property (light gray). TIP!
There’s a predefined list of colors you can use simply by typing their names. You can find the list of available colors in the W3Schools article on colors. If you want to use your own custom color, you only need to know the HEX or RGB code of the color and write it instead of LightGrey, for instance, #4679CC or rgb(7,121,204), which would result in blue, or rgba(7,121,204,0.8), which would result in the same blue but at an 80% opacity. This last one gives you the most control over your color use.

Note that CSS syntax is case-insensitive. This is true for properties (e.g., background-color or background-CoLor), as well as for values (e.g., LightGray or lightgray). Be aware, however, that the selector is case-sensitive, as this is necessary to make unique identifiers. For instance, container {..} wouldn’t be the same as Container {..}. We recommend you stick to using all lowercase whenever you write CSS, as this will prevent errors.

How to Use CSS in Your Page

Now that you’ve written a CSS rule, you need to include the CSS in your HTML in order to make it appear on your page. There are different ways to achieve this. Let’s look at them and evaluate their advantages and disadvantages!

Inline

When you take a glance at code on existing web pages, you may notice that some of them include inline CSS, or CSS written directly in HTML tags:

<body style="background-color:lightgrey;">

The main advantage of inline CSS lies in its simplicity, but keep in mind that this workflow is not recommended as it goes against the general best practice of separating content and styling. Because the styling is within the content, inline CSS also makes it very hard to find specific stylings and maintain your project when you need to make adjustments or fix bugs.

<style> Tags

Another way you can include your CSS rules is by putting them all within a dedicated tag called the style tag.

<style>
  body {
    background-color: LightGrey;
  }
</style>

The style tag is placed within the <head> tag of your HTML file. This method can be a good approach for very small one-page websites or projects.

The main disadvantage, however, is that if you need something on more than one page, you’ll need to include the same CSS code on every page. This means that if you make changes to your code, you’ll need to do that on every page, as well! This is something we usually try to avoid as it can lead to more (unnecessary) work.

External Style Sheet

This brings us to the most viable option—a dedicated file for your CSS code, also known as an external style sheet. The biggest advantage of this approach lies in its separation of content (HTML) and design (CSS) into their own respective files, which makes them easier to read and comprehend. It also allows for easy reuse of CSS rules across multiple pages. This is a great approach for websites with more than one page, making it a good choice for your portfolio site, too.

Now, let’s take a few moments to implement an external style sheet. Remember the CSS file you created earlier? This is where you’ll include all your styling rules.

To start, place the link to your CSS stylesheet within the head of your HTML files, like so:

<head>
  <link rel="stylesheet" type="text/css" href="css/style.css">
</head>

Once this is done, you can start adding your CSS rules to your “style.css” file.

Before you do that, however, it’s good practice to use a CSS Reset or Normalize File. Most browsers (or versions) have their own set of default styles for margins, borders, fonts, zoom, etc. To ensure your pages look consistent in all major browsers and that nothing will break, it’s best to start with a file that “resets” the browser styles. While there’s more than one solution for this, the most commonly used are CSS Reset), which removes everything, and Normalize CSS), which makes browsers render all elements more consistently and in-line with modern standards. It precisely targets only styles that need normalizing. For your purposes, we recommend using Normalize CSS.

There are two ways to include Normalize CSS in your portfolio site:

  1. Download the file using the link in the text above and add it to your CSS folder, then place the link to the file within the head of your HTML files just like you did with the link to your "css/styles.css" file above.

  2. Instead of downloading the file, link to a CDN (content delivery network). This saves time for those users who’ve already visited a different site on the same CDN (which is highly likely) and, thus, have already loaded the file. Use the “normalize.min.css” file from the link above.

Whichever option you choose, make sure the link is placed ABOVE the link to your “css/styles.css” file. As it will be “read” from top to bottom, your changes would be overwritten by the normalize file if you were to reverse the links. The code below shows what your <head> area should look like if you went with the CDN option to add Normalize CSS to your page:

<head>
  <link rel="stylesheet" type="text/css" href="https://cdnjs.cloudflare.com/ajax/libs/normalize/8.0.0/normalize.min.css">
  <link rel="stylesheet" type="text/css" href="css/style.css">
</head>

It’s also good practice to check your page(s) in the browser each time you make a change to check that it’s working properly. If there’s any issue, you’ll know the problem is in the most recent iteration, making it easier to find. Now that you’ve linked to the Normalize CSS file, check your site in the browser. Notice how the small space that used to be around the screen edges (which was 8px margin for the <body> element) has now been reset and is no longer visible? This looks a lot better!

Now that you know how to write a styling rule and link it to your CSS file, it’s time to practice applying basic styling such as color, borders, spacing, typography, and more via CSS. Let’s start with color!

Colors

On the web, colors are most commonly defined with hexadecimal notation (or HEX) prefixed with a hash symbol. A hex color typically consists of six characters. The hexadecimal notation of the primary color blue, for example, would be written #05676e.

Upon starting a new CSS document, make a list of all the colors you want to use within your CSS file as hex codes (see the image below). Alternatively, if you know you’ll want more control over opacity later, use the rgba value, instead. Preparing this list will make it easier to go back and simply copy-paste your color values into your CSS later, especially if you plan to use the same color for multiple properties.

Do so by creating a commented-out section at the top of your file that includes all your color codes. A comment has no effect on the styling—it’s for reference-purpose only. A comment in CSS starts with /* and ends with */.

/*
Colors used:
Primary Color (Blue): #05676e;
Secondary Color (Green): #489c79;
*/

TIP!
You can easily look up the HEX or RGB code of a color by using the pipette tool in any of your design tools (Photoshop, Sketch, etc). Web-based tools like Color Hex and Paletton can also help you find the perfect color for your website.

The image below shows a hex color picker from Adobe Photoshop (though it works the same in other software such as Sketch). Pick a color from the gradient box, then look for the HEX code in the field labelled with a “#”. Copy-paste this value into your CSS file to use that exact color in your webpage. There are also free, web-based tools like the ones mentioned above in the tip section for determining a color’s HEX code.

Typography

Specify the base styling for your typography at the top of your CSS. This should include the fonts, sizes, and spacings for your most-used elements, such as headings and paragraphs.

Embedding a Font

Let’s start with the font itself. Nowadays, it’s possible to embed a variety of different fonts directly into a website. There are many services (both paid and free) that offer different font faces for embedding. To start, you might want to check out Google Web Fonts, as it’s free and the fonts are optimized for web usage.

To embed a font:

  1. Go to Google Web Fonts.
  2. Pick a font (for this Exercise, we’ve chosen Lato as an example).
  3. Copy the embedding code into each HTML page <head> section:

Once you’ve added the embed code for a font into your HTML document, you can assign that font to any element you want via CSS. Let’s apply the font to your website:

body {
  font-family: 'Lato', sans-serif;
}

The font-family: ‘Lato’ part of the above code applies the imported font to the body of your website. What this means is that all elements within the body will automatically have the font “Lato” applied if not otherwise specified in another CSS rule. The sans-serif part is your fallback font. If, for whatever reason, Lato can’t be loaded from Google Web Fonts, your website will use the first sans-serif webfont found on the user’s computer (on Windows this is usually Arial, and on Mac, Helvetica). This prevents your website from being displayed in an unwanted font such as Times New Roman if the correct font can’t be loaded.

TIP!
Not sure where to start when selecting suitable fonts for your website? Try out fontpair.co! It’s a handy tool that allows you to pair fonts from Google Web Fonts and make sure they fit well together.

Font Weights

When selecting fonts, you’ll also want to make sure you select the appropriate font weight(s) in Google Web Fonts; otherwise, you won’t be able to make a font thin(ner) or bold(er).

The customizations shown in the above image would result in a link like this:

<link href="https://fonts.googleapis.com/css?family=Lato:300,400,700" rel="stylesheet">
Font Sizes, Line-Height, and Font Stylings

After you’ve embedded a font, you can begin styling your different text elements. Apply general text stylings to the body and more specific styling to individual elements, like this:

body {
  background-color: #ffffff;
  font-family: 'Lato', sans-serif;
  font-size: 16px; /* Sets the font size to 16px */
  font-weight: 300; /* Sets the font weight to 300 */
  line-height: 1.6; /* Sets the line-height to 1.6 * the font-size */
  color: #4a4a4a;
}

h1, h2, h3, h4 {
  line-height: 1.4;
}
h1 {
  font-size: 2.75em;
  font-weight: 700;
}

h2 {
  font-size: 2.25em;
  font-weight: 700;
}

h3 {
  font-size: 1.75em;
  font-weight: 400;
}
h4 {
  font-size: 1.35em;
  font-weight: 400;
}

Applying general stylings to the body of your HTML page is a good idea as it allows you to keep your CSS shorter. So long as your fonts are defined in your body, you don’t need to specify them for every individual element—only if you want to overwrite your default stylings (e.g., for your headings, buttons, etc.). This reduces how much code you need to write and ensures consistency across your website as every element is applied the same base styling.

A good default font size is 16px, as that’s become more or less the standard size for body copy. Avoid going much smaller than this (except for, e.g., captions or colophon).

You’ll notice that we used an em value for the other selectors instead of a px value. This is because em values are calculated using the font-size of that element’s parent, which, in this example, is the body selector. Our value for h1 is 2.75em. As the font-size for our body is 16px, this means the pixel value for our h1 will be 2.75 x 16px, or 44px. Designating font values like this makes it very simple to resize text for different screen sizes as all you need to do is change the base pixel value in your body and all the em values will update accordingly (e.g., if the base pixel value in the body of your desktop version were changed to 20px, the final value of h1 would be updated to 2.75 x 20px, or 55px). This prevents you from having to change every value manually while still allowing you to make changes where necessary.

While you can set the values for your h1 to h4 headings to whatever best suits your design, you can also use one of the “type scale calculators” you can find with a quick web search.

There are many other text stylings you can use, for instance:

  • font-weight: bold; (emboldens a font, although it’s better to use the weight as set in the font link: 300, 400, or 700)
  • font-style: italic; (italicizes a font, though make sure you selected it in Google fonts, as well)
  • font-variant: small-caps; (transforms all lowercase letters into small caps—uppercase letters set at the same height as their surrounding lowercase letters)
  • text-transform: uppercase; (transforms all letters to uppercase so that you don’t need to write all uppercase in your HTML)
  • text-decoration: underline; (underlines a piece of text)

Setting the Viewport in HTML

Before moving on, let’s discuss the first steps needed for responsive design. If you took one of our design courses previously, you should be familiar with responsive design and its mobile-first approach. This approach is also good to keep in mind when styling web pages, especially as most popular frameworks (e.g., Bootstrap, Zurb Foundation, etc.) use this approach, almost guaranteeing you’ll need to be familiar with it in your future role.

Fortunately, Chrome’s mobile view makes it easier for us to keep responsive design in mind as we work on our site. Simply open up your Developer Tools and click on the “mobile view” toggle in the top-left of the window:

From here, you can choose the mobile screen resolution you want to start with. We recommend “iPhone 5/SE” as it has the smallest screen width (320px) and, thus, the smallest viewport you need to design for.

What does your website look like now? Likely, it’s hard to read, as the browser has simply shrunk all the content to fit within the new screen size. You’ll also notice that the browser is still using your large images rather than the mobile-optimized versions you created earlier.

To fix this, you need to give the browser instructions on how to control the page scaling and dimensions. Fortunately, HTML5 comes with a function that allows you to do exactly this:

<meta name="viewport" content="width=device-width, initial-scale=1">

Including this code in your HTML will ensure that each version of your page will show correctly on all devices. Go ahead and add it to the head of each of your pages now. Afterwards, reload your page in the browser, and you should see something similar to this:

In the example above, you can see that the correct images have been loaded. This is because the browser now recognizes the viewport. The text is also easier to read; however, you might be wondering why the text doesn’t fill up the entire width of the screen the same as the images. This is because you haven’t set a width for your images yet, which is causing them to distort the layout. To make sure your images scale correctly, set a width for them in your CSS file, like this:

img {
  width: 100%;
}

When you refresh the page in your browser now, you’ll see that all the content adapts perfectly to the viewport. Of course, there’s still work to be done, but at least now you know your pages will be displayed correctly on any device.

You can read more about the viewport tag and its range of options on the W3Schools viewport page.

TIP!
You may have noticed that your headings appear a bit too large on the page. Now would be a good time to optimize the sizes of your <h1> through <h4> tags to ensure they’re more appropriate for mobile screens.

Background Colors, Borders, and Hover States

Now that you’ve acquired a basic understanding of CSS styling and the viewport, let’s go ahead and create stylings for a few base elements on your site—namely, the hyperlink and the button. Both of these elements play a big part in your brand identity and should remain consistent throughout your site.

You already learned about the different states hyperlinks can take in Exercise 1.2. Using CSS, we can further change their styling, though try not to go overboard. Hyperlinks should look like hyperlinks; otherwise, users could have a hard time navigating your site. They should also change slightly when the user hovers over or clicks on them, as this makes for a good user experience.

Let’s take a look at what your hyperlinks look like currently. On your homepage, you should have two links in your contact section:

As you can see, the Lato font has already been applied to your links thanks to the general CSS rule you already wrote, but say you wanted to change the default link color to your primary color (copied from your comments at the top of the document) and remove the text underline when hovering over the link with your mouse. You could do so like this:

a {
  color: #ff7f50;
  text-decoration: none; /* No underlining */
}
a:hover {
  color: #cc6540;
  text-decoration: underline; /* Underlining when hovering over a link */
}

Next to the hover state, you can also style the visited and active states of your links. These are what you see once you’ve already visited a link and when you actively click on a link, respectively:

a:visited { /* visited link */
  color: #ff7f50;
}
a:active { /* selected link */
  color: #994c30;
}

TIP!
a:hover must come after a:visited in the CSS definition in order to be effective! Likewise, a:active must come after a:hover in the CSS definition in order to work. Lastly, pseudo-class names, which you’ll learn more about in the following section, aren’t case-sensitive, but it’s still good practice to be consistent to prevent errors.

IMPORTANT!
There’s no hover on mobile. This is because you don’t use a mouse to navigate, but, rather, your finger. This makes it absolutely crucial that you apply an appropriate styling that alerts your users that they’ve clicked on a link (and that it’s gone into its active state). To test out your hover effects, make sure you switch back to desktop view in your Developer Tools.

Accessibility Across the Web

Because :hover is activated when the user moves their mouse over the element, it won’t do anything for users who navigate a webpage via keyboard. In order to make your website work well under all circumstances (and for all users), you should also add a :focus styling. This will be activated when an element is targeted by the keyboard. In most cases, it's good practice to use the same styling for :hover and :focus, so let’s add this:

a:hover,
a:focus {
  color: #cc6540;
  text-decoration: underline;
}

Pseudo Classes

The classes mentioned above (i.e., :hover, :visited, etc.) are called pseudo classes. They’re added via a colon after a tag name. Pseudo classes allow you to style a specific state of an element. This is important for your website accessibility. Styling these pseudo classes can help your users navigate your website more easily. For example, hover states indicate some sort of interaction is possible with an element and that it hasn’t been disabled.

Next, you’ll look into how you can style your button links, but beforehand, you first need to understand how CSS classes work.

CSS Classes

We’ve now added some styling to our base HTML elements. If you want to further expand your styling, however, you’ll probably run into a new issue. What do you do if you want two of the same type of element styled differently? For instance, let’s say you want to style certain hyperlinks as buttons. In fact, maybe you even want to have different types of buttons: a primary/default button (for your “read more” links) and a secondary button that’s styled differently (for your “email” button). The HTML syntax would be the same, but the styling would need to be different. That’s where CSS classes come in handy. You can add a CSS class to any HTML element, then apply styling to only this specific class. Let’s go ahead and add the following to make a button for your “read more” hyperlinks:

<a href="#" class="button">Read more</a>

The above code adds a new CSS class, button, to your “read more” links. With this class added in your HTML, you can then go back to your CSS and use this new button class to refer to only elements that had the button class applied. This is done by prefixing the class name with a dot and using it as a selector within your CSS style. First, let's take a look at everything we want to include in the new styling rule for your "read more" hyperlinks:

  • Its primary color as the background color
  • Bold font weight
  • A white font color
  • An inner spacing separating the button text from its border
  • Rounded corners
  • A border along the bottom

Now, take a look at the code below, which specifies the button class by using .button as the selector, then adds new styling rules for each of the points listed above:

.button {
  display: inline-block; /* since the <a> tag is inline by default, we need to change this to inline-block, but more on this in the next exercise */
  background-color: #ff7f50;
  color: #ffffff !important; /* needed to override the default link color */
  font-weight: 400;
  padding: 10px;
  border-radius: 3px;
  border-bottom: 2px solid #994c30; /* width of the border, type, color */
  transition: 0.2s opacity;
}

TIP!
Make sure you put this rule at the END of your “styles.css” document. As this is a very specific rule only for hyperlinks that have the button class, it needs to go close to the bottom; otherwise, it could get overwritten by other less specific rules, such as the general styling rules for all of your hyperlinks.

Don't forget to give your button some styling rules for its hover state, as well! For instance, it might be nice to have your button “light up” slightly upon hover. You can also change the underline to appear in the link when hovering (the same as you set for :hover previously). Select the hover state of your button class the same way you did above—by adding :hover to the end of your button selector:

.button:hover {
  opacity: 0.8; /* Reduces the opacity of the button to 80% */
  text-decoration: none;
}

IMPORTANT!
Just like with hyperlinks, the hover styling won’t have any effect on mobile (and you probably won’t see this change in the Web Inspector when viewing the mobile version). Instead, just as you did for your hyperlinks, make sure you have styling for the .button:active state, as this will ensure the action is visible when selecting the button.

Now, let’s say you want your “email” button to be styled differently than the other buttons. To do this, add the button class to the hyperlink in HTML like you did before, but this time, add an additional second class to make sure the different styling is only applied to this button alone (or any other elements you also want to be styled in this particular way).

<a href="mailto:office@careerfoundry.com" class="button contact">Send me an email</a>

In the code above, the class “contact” has been added in addition to the “button” class. Now, look at your CSS, where you’ll want to add a new styling for the “button contact” class combination. As you can see below, you’ll want to designate them one after another without a space in between (.button.contact):

.button.contact {
  background-color: transparent;
  color: #ff7f50 !important;
  text-transform: uppercase;
  border: 2px solid #ff7f50;
}

Before moving on, don’t forget to add the hover action (and active state), as well:

.button.contact:hover {
  background-color: #ff7f50;
  color: #ffffff !important;
}

As you can see, only the properties and values that change are added to the class. The button will already have adopted the style of the first button we listed above. This is where the cascading behavior of CSS comes into play. The rules listed first in the stylesheet are applied first, then the next rules, then the next, all the way to the bottom, adding onto each other or replacing elements of each other where applicable.

As you’ve seen above, HTML elements can have any amount of class names. You can assign multiple class names by separating them with spaces. It's considered best practice to use lowercase class names separated with hyphens; for example, an element could have the following class names: button-secondary button-small. Here, both the .button-secondary, as well as the .button-small rule-sets would apply to this element.

Selectors

As you learned at the beginning of this Exercise, selectors are part of the CSS rule-set. Here, again, as a reminder:

With the selector, you specify the HTML element you want to style with CSS. Selectors can be as simple as the name of the HTML tag, like body, or as complex as #hero > .my-content:hover. Let’s have a look at the different options:

  • Simple selectors will select all elements based on element type, class, or ID. For instance: a, #my-element, or .my-element
  • Attribute selectors will select all elements based on their HTML attributes/attribute values. For instance, if you had a text-input element, you could select it with: input[type=”text”]
  • Pseudo-classes will select all elements that exist in a certain state, such as an element that’s being hovered over by the mouse pointer, a checkbox that’s currently disabled or checked, or an element that’s the first child of its parent in the DOM tree. You’ve already used a pseudo-class selector earlier in this Exercise when styling the hover state of your links: a:hover.
  • Pseudo-elements will select all content that’s in a certain position in relation to an element—for example, the first element of a list, which could be selected via li:first-child.
  • Combinators, while not exactly selectors themselves, are ways of combining two or more selectors in useful ways for very specific selections. For example, you could select only paragraphs that come directly after headings to give them more spacing. To select such paragraphs, you can use h1 + paragraph.
  • Multiple selectors can be placed in the same CSS rule, separated by commas, to apply a single set of declarations to all the elements, for instance: header, .header, .header-primary {}

Check out the following articles to learn more about the different selector types: Simple selectors, Attribute selectors, Pseudo-classes and pseudo-elements, Combinators and multiple selectors.

Specificity

Another thing you should be aware of when writing CSS is the specificity. Especially in the beginning, you may find yourself overwriting your own CSS rules without realizing it due to a lack of specificity awareness. Specificity is a measure of how specific a selector is—in other words, how many elements it could match. Element selectors (such as buttons), for example, have low specificity. Class selectors, on the other have, have a higher specificity, so they'll win out. ID selectors (starting with a #) have an even higher specificity, so they'll win out against class selectors. The only way to win against an ID selector is to use !important. Let’s look at an example:

This is how the link would look in HTML:

<a href="https://www.google.com" class="my-link" id="my-link">A link</a>

And as for the CSS, we would add:

a {/* element selector */
  color: red;
}

#my-link {/* id selector */
  color: green;  
}

.my-link {/* class selector */
  color: blue;  
}

Now, based on the information you’ve read before, which color do you think the link will have? Red, green, or blue?

The right answer is green. This is because the ID selector (#my-link) wins out over the class and the element selector. Be aware of that when writing your CSS. In general, it's good practice to rely heavily on class selectors and element selectors and less on ID selectors (also, the ID selector can only be used for one unique element on a page). If two selectors have the same specificity, the one that comes later in the stylesheet will win out.

Check out more examples in the Mozilla documentation on specificity.
